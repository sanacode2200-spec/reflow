RehabCalendar 仕様書

FullCalendar を使わない自前の療法士列×時間軸グリッド実装。ReFlow への移植を前提とした検証コンポーネント。

✓ グリッド描画 ✓ 予約表示 ✓ ドラッグ&ドロップ ✓ 繰り返し展開 ✓ 詳細パネル ✓ 複数日コピー ✓ 重複検知 ✓ コールバックAPI △ 削除UI △ 新規作成UI △ 繰り返し編集ダイアログ

概要

週単位で療法士ごとの予約を表示・操作するカレンダーコンポーネント。FullCalendar を使わず Tailwind CSS + dnd-kit で自前実装。

項目仕様
表示単位週表示（月〜土の6日間）
時間軸08:00 〜 18:00
スロット刻み20分（= 40px）
グリッド高さ1200px（30スロット × 40px）
列単位療法士ごと（複数同時表示可）
列幅88px / 列、時間軸列 52px

技術スタック

ライブラリ用途バージョン要件
Next.js App Routerフレームワーク15系
TypeScript型安全strict + noUncheckedIndexedAccess
Tailwind CSSスタイリングv4系
dnd-kitドラッグ&ドロップ@dnd-kit/core, @dnd-kit/utilities
date-fns日付操作v3系
移植先に dnd-kit が入っていない場合はインストールが必要。Tailwind のバージョンが v3 の場合は一部クラス記法が異なる可能性あり。

型定義

lib/types.ts に定義。ReFlow のスキーマに準拠しており、移植時に既存型と統合・流用可能。

入力データ型

type Occupation = 'pt' | 'ot' | 'st'

type Staff = {
  id: string
  name: string
  occupation: Occupation
}

type Patient = {
  id: string
  name: string
}

type Schedule = {
  id: string
  patient_id: string
  therapist_id: string
  start_at: string          // ISO8601 ローカル時刻 "2026-05-18T09:00:00"
  end_at: string            // ISO8601 ローカル時刻
  recurrence_rule: string | null  // iCalendar RRULE形式 or null
}

内部型（展開済みインスタンス）

type ScheduleInstance = {
  id: string            // 単発: schedule.id / 繰り返し: "{schedule_id}__{yyyy-MM-dd}"
  schedule_id: string   // 元の Schedule.id
  patient_id: string
  therapist_id: string
  start_at: Date        // Date オブジェクト（展開済み）
  end_at: Date
  is_recurring: boolean
}
Schedule は文字列日時（DB保存形式）、ScheduleInstance は Date オブジェクト（描画用）。コンポーネント内部で expandSchedules() が変換する。

Props

エントリポイント RehabCalendar の公開 Props。

プロパティ 型 必須 説明
staffs Staff[] ✓ 全スタッフ一覧。D&Dの列インデックス維持に使うため、表示していないスタッフも含めて渡す。
patients Patient[] ✓ 全患者一覧。予約ブロックと詳細パネルの名前表示に使用。
schedules Schedule[] ✓ 予約データ。controlled: 親が管理し、コールバック経由で更新する。
currentStaffId string ✓ ログイン中のスタッフID。初期表示列の決定と削除防止（自分の列は外せない）に使用。
initialWeek Date — 初期表示週の任意の日付。省略時は今週。内部で月曜日に正規化される。

コールバック

すべてオプション。渡さない場合は操作しても UI に反映されない（controlled のため）。

コールバック 型 発火タイミング
onScheduleUpdate (schedule: Schedule) => void ドラッグ&ドロップ完了時。変更後の Schedule オブジェクトを渡す。
onScheduleCreate (schedules: Schedule[]) => void 複数日コピー確定時。新規作成分の Schedule 配列を渡す（1件でも配列）。
onScheduleDelete (scheduleId: string) => void 削除操作時（削除UIは未実装。移植先で実装後に繋ぐ）。

接続例

// 最小構成（Next.js App Router, 'use client' が必要）
'use client'
import { useState } from 'react'
import RehabCalendar from '@/components/features/schedule/RehabCalendar'

export default function SchedulePage() {
  const [schedules, setSchedules] = useState<Schedule[]>(initialData)

  return (
    <RehabCalendar
      staffs={staffs}
      patients={patients}
      schedules={schedules}
      currentStaffId={session.staffId}
      onScheduleUpdate={(updated) => {
        setSchedules(prev => prev.map(s => s.id === updated.id ? updated : s))
        api.patch('/schedules/' + updated.id, updated)  // ReFlowのAPI
      }}
      onScheduleCreate={(created) => {
        setSchedules(prev => [...prev, ...created])
        api.post('/schedules/bulk', created)
      }}
      onScheduleDelete={(id) => {
        setSchedules(prev => prev.filter(s => s.id !== id))
        api.delete('/schedules/' + id)
      }}
    />
  )
}

ヘッダー構造

ヘッダーは2段構成で sticky 表示される。

行内容z-index
1行目曜日 + 日付（今日はハイライト）20
2行目スタッフ名 + 職種バッジ（PT/OT/ST）20

グリッド線

  • 1時間ごと（3スロットごと）: #d1d5db（濃い線）
  • 20分ごと: #f3f4f6（薄い線）
  • 曜日境界: 左端に 1px solid #d1d5db

実装済み機能

スタッフ列の表示管理

  • 初期表示: currentStaffId の列のみ
  • ナビバーの「+ 追加」ドロップダウンから他スタッフを追加可能
  • 自分の列（currentStaffId）は削除不可、他スタッフは ×ボタンで非表示
  • 表示中スタッフは職種別カラーチップ（PT=青、OT=緑、ST=紫）で表示

週ナビゲーション

  • 「前週 / 翌週 / 今週」ボタン
  • 今日の列は薄い青背景でハイライト

予約ブロック（EventBlock）

  • top・height を px 計算して position: absolute で配置
  • 職種で色分け（PT=青、OT=緑、ST=紫）
  • 高さ 32px 以上の場合のみ時刻（HH:mm–HH:mm）を表示
  • 繰り返し予約は右上に黄色ドット表示
  • ドラッグ中はゴースト（破線枠）を元の位置に表示

詳細パネル（DetailPanel）

  • 予約クリックで右サイドに 288px のパネルをスライドイン
  • 表示内容: 患者名・担当療法士・日時・繰り返し有無・所要時間
  • 「複数の日付にコピー」ボタンから CopyDatePicker を開く

繰り返し予約

対応フォーマット（lib/recurrence.ts）

iCalendar RRULE の BYDAY 部分のみ解析。Phase 1 では月〜土の曜日固定パターンのみ対応。

// 対応例
"FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR,SA"  // 月〜土
"FREQ=WEEKLY;BYDAY=MO,WE,FR"            // 月・水・金
"FREQ=WEEKLY;BYDAY=TU,TH"              // 火・木

展開ロジック（expandSchedules）

表示週の月〜土の6日間を走査し、BYDAY に該当する曜日に繰り返しインスタンスを生成。

expandSchedules(schedules: Schedule[], weekStart: Date, weekEnd: Date): ScheduleInstance[]

// インスタンスID形式
// 単発: schedule.id
// 繰り返し: "sched-1__2026-05-18"
ドラッグ&ドロップで繰り返し予約を移動すると、onScheduleUpdate に recurrence_rule を含む元の Schedule が渡される（全週に影響する）。「この日だけ変更」ダイアログは未実装。移植先で実装が必要。

重複検知（hasConflictOnDate）

hasConflictOnDate(
  schedules: Schedule[],
  therapistId: string,
  excludeScheduleId: string,  // コピー元を除外
  copyStart: Date,
  copyEnd: Date,
): boolean

単発・繰り返しの両方に対応。コピー先の日付に同スタッフの時間重複がある場合 true を返す。

ドラッグ&ドロップ

ライブラリ

@dnd-kit/core。PointerSensor を使用、移動距離 4px 以上でドラッグ開始（誤操作防止）。

ドロップゾーンID

各セルは col-{dayIdx}-{staffIdx} 形式の ID を持つ。

// dayIdx: 0=月 〜 5=土
// staffIdx: props.staffs 配列での元インデックス（表示非表示に関係なく不変）
// 例: col-2-0 = 水曜日・staffs[0]の列

移動計算

// 垂直方向: ドラッグ距離をスロット単位にスナップ
deltaMin = Math.round(delta.y / SLOT_HEIGHT_PX) * SLOT_MINUTES

// 範囲クランプ（グリッド外に出ない）
clampedMin = Math.max(START_HOUR * 60, Math.min(END_HOUR * 60 - duration, rawMin))

// 水平方向: ドロップ先の colId から曜日・スタッフを取得して適用

DragLayer（オーバーレイ）

ドラッグ中はカーソルに追従する半透明プレビューを DragOverlay で表示。元の位置にはゴーストを残す。

複数日コピー（CopyDatePicker）

フロー

  1. 詳細パネルの「複数の日付にコピー」ボタンをクリック
  2. ミニカレンダーモーダルが開く
  3. コピー先の日付を複数選択（コピー元の日付は選択不可）
  4. 「コピー」ボタンで確定 → onScheduleCreate 発火

重複表示

状態表示
コピー元の日付グレー・選択不可
重複あり・未選択オレンジ枠・テキスト
重複あり・選択済みアンバー塗りつぶし
重複なし・選択済みスカイブルー塗りつぶし

重複がある月はカレンダー下部に凡例を表示。フッターに「N日選択中（うちM件は重複）」を表示。

コピー時の処理

// コピー元と同じ時刻・所要時間・スタッフ・患者を引き継ぐ
// idは "copy-{元schedule_id}-{timestamp}-{index}" で生成
// recurrence_rule は null（単発として追加）

組み込み手順

1. ファイルをコピー

# このリポジトリから cp -r components/rehab-calendar/ {移植先}/components/features/schedule/ cp lib/grid.ts {移植先}/lib/grid.ts cp lib/recurrence.ts {移植先}/lib/recurrence.ts cp lib/types.ts {移植先}/lib/types.ts # 既存型と統合する場合は要調整

2. 依存パッケージの確認

npm install @dnd-kit/core @dnd-kit/utilities date-fns

3. コールバックを API に繋ぐ

上記「接続例」を参照。onScheduleUpdate / onScheduleCreate / onScheduleDelete の3本を移植先の API クライアントに繋ぐ。

4. FullCalendar との置き換え

FullCalendar のイベントは { title, start, end, ... } 形式が多い。Schedule 型（patient_id, therapist_id, start_at, end_at）に変換するアダプターが必要になる場合がある。

5. 確認ポイント

  • 移植先の Tailwind バージョン（v3 の場合、一部ユーティリティクラスが異なる）
  • @/lib/... のパスエイリアスが移植先でも有効か
  • 移植先で dnd-kit の DndContext が重複しないか（既に使っている場合は統合が必要）
  • start_at / end_at のタイムゾーン。現在はローカル時刻前提。UTC管理の場合は変換が必要。
